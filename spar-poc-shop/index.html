How the SSO Flow Works
1
User logs in on Site A
CDC Screen-Set on kingcode-software.github.io/spar-poc/ authenticates the user and sets a session cookie on cdns.eu1.gigya.com.
2
User navigates to Site B (this page)
No login prompt is shown. gigya.js loads with Site B's API key — both sites belong to the same CDC Site Group.
3
Site B silently checks the CDC session
accounts.getAccountInfo() is called on page load. CDC checks the shared group session cookie and returns the user's identity — no credential prompt.
4
SSO confirmed — user is recognised
Site B receives the UID, email and profile from CDC. The user is shown as authenticated. This demonstrates production-like cross-site SSO via the SAP CDC Site Group.
Architecture Notes
⚠️ Third-Party Cookie Caveat (POC Limitation) CDC cross-site SSO relies on a session cookie set on cdns.eu1.gigya.com. Modern browsers (Safari ITP, Firefox ETP) may block this as a third-party cookie. In production this is mitigated via SAP CDC Certificate Provisioning + Custom API Domain, routing CDC calls through your own domain so the cookie is treated as first-party. This is analogous to how WSO2 removes the CDC domain-bridging step entirely.
🔵 OIDC Isolation Site B has no OIDC clients registered. The OIDC Authorization Code + PKCE flow remains entirely on Site A. Site Group membership for SSO does not affect or interfere with the existing OIDC configuration.
